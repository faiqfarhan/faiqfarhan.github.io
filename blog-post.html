How I cut automation test time by 30% at InstaFreight (without rewriting everything)

When a test suite takes longer to run than a sprint, it stops being useful. Here's the exact changes I made to InstaFreight's Postman/Newman setup that delivered a 30% efficiency gain — without throwing away any existing work.

FF
Faiq Farhan
R&D Verification Engineer II at Ansys · ex-InstaFreight QA Analyst
March 12, 2025 · 9 min read

When I joined InstaFreight in January 2022, the QA process was working — in the sense that bugs were being found and software was being shipped. But the automated testing setup had grown organically without much structure, and it was showing. Test runs were slow, results were inconsistent, and the team had developed a habit of re-running tests twice "just to be sure."

By the time I left in mid-2023, we had a 30% improvement in testing efficiency, a CI pipeline that developers actually trusted, and a test coverage map that product managers could actually read. Here's exactly what changed.

The starting point: what was broken

InstaFreight's core product is a freight-matching platform — shippers post loads, carriers bid, the platform matches them. The API layer is the real product: a large REST API covering shipment creation, carrier discovery, pricing, bidding, and tracking events.

The existing Postman collection had grown to around 150 requests organised loosely by "what someone was testing that day." There was no consistent naming, no environment separation, and authentication tokens were hardcoded — meaning tests broke every time someone's token expired. Tests were run manually by whoever was doing QA that sprint, results were screenshots in a Confluence page, and there was no CI integration at all.

The real problem wasn't the tooling — Postman and Newman are perfectly capable of handling this properly. The problem was that the collection had grown without a structure to grow into.

Step 1: audit and restructure the collection

Before writing a single new test, I spent a week going through the existing collection request by request. I categorised every request into one of four buckets:

  1. Keep as-is — well-written, clear assertions, testing something important
  2. Fix and keep — testing the right thing but with hardcoded values or missing assertions
  3. Consolidate — duplicate tests covering the same scenario from slightly different angles
  4. Delete — outdated tests for deprecated endpoints, or tests with no assertions that were effectively just "this request doesn't 500"

About 35% of the collection fell into categories 3 and 4. Removing or merging those immediately made the suite faster to run and easier to understand.

I then reorganised the remaining requests into folders matching the API domain: Authentication, Shipments, Carriers, Pricing, Bidding, Tracking. Each folder had a consistent naming convention: [METHOD] endpoint — scenario description. For example: POST /shipments — create shipment with valid payload and POST /shipments — reject shipment with missing origin address.

Step 2: fix authentication properly

Hardcoded auth tokens were the biggest source of test failures that weren't actually bugs. The fix was straightforward but required changing how every test in the collection worked.

I added a pre-request script at the collection level that checks whether a valid token exists in the environment and fetches a new one if not:

// Collection-level pre-request script
const token = pm.environment.get('auth_token');
const expiry = pm.environment.get('token_expiry');
const now = Date.now();

if (!token || now > parseInt(expiry)) {
    pm.sendRequest({
        url: pm.environment.get('base_url') + '/auth/token',
        method: 'POST',
        header: { 'Content-Type': 'application/json' },
        body: {
            mode: 'raw',
            raw: JSON.stringify({
                client_id: pm.environment.get('client_id'),
                client_secret: pm.environment.get('client_secret')
            })
        }
    }, (err, res) => {
        pm.environment.set('auth_token', res.json().access_token);
        pm.environment.set('token_expiry', now + (3600 * 1000));
    });
}

This alone eliminated a huge category of false failures. Tests stopped breaking because someone's token had expired the previous night.

Step 3: data-driven testing with CSV files

A lot of the API behaviour depended on the specific data being sent — different shipment sizes triggered different pricing rules, different origin/destination combinations matched different carrier pools. The original tests covered maybe 5 data variations manually. We needed more without writing 5× as many tests.

I created CSV data files for the high-value scenarios and ran Newman with the --iteration-data flag. For the shipment creation endpoint alone, this let us cover 40 data variations with a single test request.

# Running Newman with data file in CI
newman run collections/shipments.postman_collection.json \
  --environment environments/staging.postman_environment.json \
  --iteration-data test-data/shipment-scenarios.csv \
  --reporters cli,junit \
  --reporter-junit-export results/shipments-results.xml

The outcome

The combination of removing redundant tests, fixing authentication, parallelising Newman runs per domain, and adding data-driven coverage got us to the 30% efficiency figure. But the number I was more proud of was the reduction in re-runs — from "always run it twice" to "re-run only when there's a genuine reason."

That shift happened because developers started trusting the results. When a test failed, it was because something in the application had changed, not because the test infrastructure was unreliable. That trust is the actual goal — efficiency is just the measurable proxy for it.

The full Newman + GitHub Actions setup I built from these patterns is available as an open-source project: see the Selenium POM Framework and JMeter Load Test Suite on my projects page. The same principles apply across both.
